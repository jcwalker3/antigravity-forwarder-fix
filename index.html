‚ö° The Error

When connecting to a dev container in Antigravity IDE, the IDE crashes with the following error logged in the output panel:

‚úñ
forwarder error: handleClient Error: subprocess terminated immediately with return code 127

Exit code 127 in Unix means "command not found". The IDE is trying to execute a node binary inside the container, but the path it constructed doesn't actually exist on the filesystem.

üîç Root Cause Analysis

The forwarder.js script is responsible for setting up a TCP/socket bridge between your local machine and the dev container. It does this by running a Node.js process inside the container via docker exec.

The critical problem is in the handleClient() method at line 297. The IDE provides a remoteServerNodePath ‚Äî the absolute path to the Node binary inside the container ‚Äî which looks something like:

Example path structure 
/home/user/.antigravity-server/bin/abc1234/node
                                              ‚Üë
                                     commit hash prefix

When the IDE updates on the host side, the commit hash prefix in the cached path changes ‚Äî but the container's filesystem still has the old version installed. The original code uses this path verbatim, so bash can't find the binary and returns exit code 127.

IDE expects
/home/user/.antigravity-server/bin/abc1234/node
‚Üí
Container has
/home/user/.antigravity-server/bin/def5678/node
üí° Key Insight

The node binary exists in the container ‚Äî just under a different commit hash directory. The structure and filename are identical; only the version-specific directory name has drifted out of sync.

üìú The Buggy Code

The original handleClient() method naively concatenates the node path supplied by the IDE, trusting it to be correct:

forwarder.js ‚Äî lines 297‚Äì299 Before 
- nodeCommand = ''
-     .concat(remoteServerNodePath, ' -e "')

‚ú® The Fix ‚Äî Universal Path Resolver

Instead of trusting the exact path, the fix dynamically discovers the correct node binary by glob-matching the parent directory:

forwarder_patched.js ‚Äî lines 297‚Äì306 After 
// Universal Path Resolver - handle version prefix mismatch
+ var pathParts = remoteServerNodePath.split('/');
+ var nodeBin = pathParts[pathParts.length - 1];
+ var commitDir = pathParts[pathParts.length - 2];
+ var binDir = pathParts.slice(0, -2).join('/');
+ nodeCommand = 'NODE_PATH=$(ls -d '
+     .concat(binDir, '/*')
+     .concat(commitDir, ' 2>/dev/null | head -1)
+     && \"${NODE_PATH}/')
+     .concat(nodeBin, '\" -e \"')
+     .concat(nodeJsCode, '\"');

How It Works

üß©

1. Decompose the Path

Splits the expected path into three parts: the base directory, the commit hash directory, and the node binary filename.

üîé

2. Glob Search

Uses ls -d with a wildcard to find any directory matching the pattern ‚Äî regardless of which commit hash is installed.

üöÄ

3. Dynamic Execution

Stores the discovered path in $NODE_PATH and uses it to execute the node binary ‚Äî always finding the right one.

The resulting shell command becomes:

# Glob finds the actual commit hash dir, whatever it is
NODE_PATH=$(ls -d /home/user/.antigravity-server/bin/* 2>/dev/null | head -1) \
  && "${NODE_PATH}/node" -e "...code..."
